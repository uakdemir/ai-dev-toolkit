- [ ] **Step 2: Append monorepo action types section**

After the existing content, add a new section `## Monorepo Migration Action Types` with these subsections:

1. **`create-package`** — Create package directory + manifest with placeholder dependencies. Include per-stack templates:
   - Node.js: `package.json` with name, version, main fields + TODO comment for dependencies
   - Python: `pyproject.toml` with project name, version + TODO comment
   - .NET: `.csproj` with target framework + TODO comment for PackageReference
   - Header comment: `{comment_char} Generated by implement-plan. TODO: Review and verify dependencies.`

2. **`create-workspace-config`** — Generate workspace config at monorepo root. Templates:
   - pnpm-workspace.yaml: `packages: ['packages/*', 'shared/*']`
   - npm/yarn: `"workspaces": ["packages/*", "shared/*"]` in package.json
   - Python: conditional — hatch (pyproject.toml workspace), pants (pants.toml), namespace packages (skip with note)
   - .NET: add .csproj to existing .sln

3. **`rewrite-cross-package-import`** — Update imports to use workspace package names.
   - Serena mode: `find_referencing_symbols` to find all consumers, update import paths
   - Regex fallback: scan for import statements targeting moved files, replace with new package path (e.g., `../auth/service` → `@myorg/auth`)

4. **`update-config`** — AI-judgment step. Per-stack examples:
   - Node.js: add `paths` to tsconfig.json, update jest.config.js module mapper
   - Python: update pyproject.toml dependency paths
   - .NET: add `<ProjectReference>` to consuming .csproj files
   - Note: no deterministic algorithm — AI interprets migration plan instructions

5. **`[DONE]` marker for monorepo** — Phase headings: `## [DONE] Phase N:`. Within-phase action groups: `[DONE] Files to move:`.

- [ ] **Step 6: Add Step 3 monorepo parsing (alongside existing layer parsing)**

When format is monorepo, parse using keyword matching:
- Identify phases: scan for `## Phase N:` headings or `**Phase N:**` bold markers (also match `## [DONE] Phase N:`)
- Phase 0: extract by keywords `workspace`/`shared library`/`files to move`/`import paths`
- Phases 1-N: extract module name, files to move, imports to update, config changes, tests to verify — all by keyword matching
- Internal data model: phase_number, phase_name, module_name, target_package_path, files_to_move, imports_to_update, config_changes, verification, status
- Validation: Phase 0 must exist, Phase 1-N must have files

- [ ] **Step 7: Add monorepo pre-flight validation to Step 5**

For monorepo plans: target package dirs shouldn't exist (unless resuming), workspace root writable, source files exist, no cross-phase collisions. On failure: "abort or adjust plan" (no skip).

- [ ] **Step 8: Add verification granularity mapping to Step 6**

For monorepo: per-step = after each file move; per-phase = after each monorepo phase (Phase 0, 1, 2...); end-only = after all phases. Default recommendation: per-phase.

- [ ] **Step 9: Add monorepo execution path to Step 7**

This is a parallel fork — when plan type is monorepo, execute Phase 0 then Phases 1-N with action-type grouping within each phase:
- Phase 0: create-workspace-config → create-package (shared) → move-files → rewrite-cross-package-import → verify
- Phases 1-N: create-package → move-files → rewrite-cross-package-import → update-config → verify
- CI setup: output instructions, don't generate config
- Do not proceed to Phase N+1 until Phase N's checkpoint passes

- [ ] **Step 10: Add commit strategy section**

One commit per monorepo phase:
- Phase 0: `refactor: prepare monorepo workspace — extract shared library (Phase 0)`
- Phase N: `refactor: extract {module_name} to packages/{module_name} (Phase {N})`
- Partial: `refactor: extract {module_name} (Phase {N}, partial — {action_type} complete)`

- [ ] **Step 11: Add resume strategy section**

Phase-level: `## [DONE] Phase N:` markers on headings. Within-phase: `[DONE]` on action group headers. Filesystem reconciliation: create-package (dir exists → skip), move-file (source gone + target present → skip), rewrite-cross-package-import (re-scan), create-workspace-config (exists → skip; intentionally skipped → "not applicable").

- [ ] **Step 12: Add monorepo error handling rows**

Add to existing error handling table:
- Unrecognized plan format
- Monorepo missing sources (abort or adjust)
- Package dir already exists (resume or abort)
- Workspace config already exists (update or skip)
- Cross-package rewrite fails (flag, user confirms)
- Phase verification fails (stop, no Phase N+1)
- CI setup needed (instructions only)
- Unparseable phase (stop, user fixes, resume)
- Config instruction ambiguous (present to user)
